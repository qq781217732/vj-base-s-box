# 类潜兵开放地图遭遇战模式落地实现方案

日期：2026-05-02  
项目：`f:/DevProject/Sbox/npc_zombie`  
范围：只面向新模式设计，不继续兼容或扩展原有 `SurvivalGamemode` 玩法逻辑。

---

## 1. 文档目标

本文档定义一套可落地的新模式架构，用于实现接近《绝地潜兵》体验的开放地图遭遇战系统。目标不是简单增加刷怪量，而是建立一套能稳定支持以下体验的上层框架：

1. 大地图被划分为多个遭遇区域。
2. 不同区域在平时维持游荡敌人和小规模巡逻。
3. 玩家在区域内推进、开火、爆炸、完成任务时，会触发局部升温和波次增援。
4. 地图层面存在持续压力，玩家越高调、越拖时间、越深入高危区域，越容易被持续追逼。
5. 敌人不是全图无脑锁定玩家，而是通过感知、警戒传播、区域升级、增援编排，形成逐步扩大的遭遇战。

本方案默认：
- 新模式独立于旧的全局波次生存模式。
- 旧代码只复用底层移动/导航/实体能力，不复用旧导演和旧模式节奏。
- 本文档以“可开发实现”为目标，包含模块拆分、运行流程、配置模型与迭代顺序。

---

## 2. 新模式的核心体验定义

### 2.1 玩家体验目标

玩家进入地图后，应持续感受到以下节奏：

1. **平时有压迫，但不是全图开战**  
   区域内存在游荡敌人、局部巡逻、零散威胁。

2. **战斗会扩散，而不是瞬间全开**  
   一只怪发现玩家，不代表全图都同时冲来；但如果玩家持续开火、引爆、滞留，会引来越来越多的敌人。

3. **任务会制造峰值战斗**  
   启动装置、护送、占点、防守、采集等任务会主动触发波次和增援。

4. **地图本身是活的**  
   即使没有正在进行的任务，导演系统也会根据玩家行为、地图压力、区域状态，把敌人持续布进玩家前方、侧翼、后方或相邻区域。

5. **不同敌人有基础差异，但共享统一遭遇框架**  
   普通怪、重甲怪、冲锋怪都在同一套感知/警戒/刷怪系统下运作，只在参数和战斗职责上不同。

### 2.2 非目标

本方案暂不覆盖：
- PvP
- 复杂剧情脚本
- 动态地形破坏
- 完整任务内容编辑器
- 旧生存模式与新模式共存时的无缝切换细节

## 3. 现有代码的复用与放弃范围

### 3.1 复用部分

以下旧系统作为底层复用：

1. `BaseZombie` 及其移动、转向、攀爬、寻路执行能力。
2. `NavSteer`、`NavPath` 作为导航执行层。
3. 现有敌人实体类型：`CommonZombie`、`UncommonZombie`、`ChargerZombie`。
4. 现有受击、近战、动画、音效、模型系统。
5. `HammerSpawnBlocker` 这一类“刷怪点约束”的地图工具思路。

这些能力仍然有价值，因为它们解决的是“怪怎么动、怎么打、怎么被击中”的底层问题。

### 3.2 不再作为新模式核心的旧系统

以下旧系统不继续承担新模式上层玩法职责：

1. `GameDirector`：不再作为新模式的主导演。
2. `SurvivalGamemode`：不再作为新模式的游戏节奏框架。
3. `CommonZombie` 里当前的上层 AI 逻辑（如固定概率连锁、简单追击切换）不再继续堆功能。

结论：**保留底层，替换上层。**

## 4. 总体架构

新模式拆成 5 层：

1. **Mode Layer**：新模式壳层，负责模式启动与总配置。
2. **Director Layer**：导演层，负责全局压力、区域激活、遭遇预算、增援调度。
3. **Zone / Encounter Layer**：区域与遭遇层，负责局部状态、任务联动、刷怪源管理。
4. **Perception / Alert Layer**：感知与警戒传播层，负责视线、近距、声音、伤害、同伴提醒。
5. **Pawn Layer**：敌人实体执行层，负责移动、攻击、动画、受击。

### 4.1 模块关系

- `EncounterGamemode` 持有全局配置与运行入口。
- `EncounterDirector` 管理所有区域、活跃遭遇与地图压力。
- `EncounterZone` 表达地图上的一个可独立升温的区域。
- `EncounterInstance` 表达一次具体遭遇（巡逻、警戒、任务波次、增援潮）。
- `SpawnSource / SpawnRule / SpawnGroup` 控制刷怪从哪里来、何时能来、一次来什么。
- `ZombieBrain` 决定单体敌人的感知、目标与状态切换。
- `BaseZombie` 负责执行 Brain 给出的移动/攻击结果。

## 5. 目录与类职责建议

建议新增目录：

```text

  AI/
    ZombieBrain.cs
    ZombiePerception.cs
    ZombieAlertModel.cs
    ZombieTask.cs
    NoiseSystem.cs
    NoiseEvent.cs
  Data/
    EncounterZoneConfig.cs
    EnemyArchetypeConfig.cs
    EncounterTemplateConfig.cs
    DirectorConfig.cs
```

### 5.1 `EncounterGamemode`

职责：
- 新模式启动入口。
- 装配 `EncounterDirector`。
- 持有全局配置引用（难度、敌人预算倍率、玩家人数倍率等）。
- 定义新模式生命周期（地图开始、任务开始、撤离阶段、失败结算等）。

原则：
- 不负责具体刷怪。
- 不负责单体 AI。
- 不直接操作区域内的怪。

### 5.2 `EncounterDirector`

职责：
- 管理全图压力值。
- 管理全图活跃遭遇上限。
- 维护“当前哪些区域处于升温/交战/冷却”。
- 决定是否派出巡逻、增援、追击队。
- 在玩家行为和区域状态变化时调整遭遇预算。

它是新模式的真正总导演。

### 5.3 `EncounterZone`

职责：
- 表示一个地图区域。
- 持有自己的刷怪源列表、区域预算、危险等级、任务绑定、冷却时间。
- 跟踪该区域的玩家数量、当前敌人数量、区域噪声、区域警戒值。
- 根据导演命令进入不同状态。

### 5.4 `EncounterInstance`

职责：
- 表示一次具体的遭遇对象。
- 一个区域可以有多个 Encounter，但同一时间只有少量活跃实例。
- 负责记录：
  - 类型（巡逻 / 增援 / 伏击 / 任务波次 / 追击）
  - 当前阶段
  - 已刷敌人列表
  - 是否完成 / 失败 / 过期

### 5.5 `SpawnSource`

职责：
- 表示一个可被导演使用的刷怪来源点。
- 支持不同类型：
  - 区域内部隐蔽点
  - 地图边缘入口
  - 建筑内部
  - 洞口/下水道
  - 专用增援入口
  - 任务点专用入口

### 5.6 `SpawnRule`

职责：
- 判断某个 `SpawnSource` 当前是否可用。
- 规则包括：
  - 是否在玩家视线外
  - 是否距玩家太近
  - 是否属于当前激活区域
  - 是否只允许任务事件使用
  - 是否只允许高压时使用

### 5.7 `SpawnGroup`

职责：
- 表示一次刷怪的编排单位。
- 一次刷怪不是单体 `new CommonZombie()`，而是一个具备意图和组成的组。

示例：
- 巡逻组：`3 Common + 1 Uncommon`
- 小型骚扰组：`4 Common`
- 任务增援组：`6 Common + 1 Charger`
- 高压追击组：`2 Fast + 4 Common + 1 Armor`

### 5.8 `ZombieBrain`

职责：
- 管理单体敌人的决策状态。
- 接收感知事件与导演命令。
- 选择当前目标与当前任务。
- 将“追谁、去哪里、调查哪里、是否脱战”转换为底层执行命令。

原则：
- 身体和大脑分离。
- `CommonZombie` 不再同时承担实体和完整 AI 决策。

## 6. AI 状态分层设计

不能继续只用旧的 `Wander / Chase / Lure / Burning` 单层状态。新模式采用三层模型。

### 6.1 警戒层（Awareness）

定义敌人对环境和玩家的认知程度：

1. `Dormant`：休眠，几乎不主动响应，适合埋伏点或远区低开销怪。
2. `Idle`：正常巡逻/游荡。
3. `Suspicious`：察觉异样，转向、驻足、提高感知。
4. `Investigating`：已获得大致方向，前往声音点或最后目击点。
5. `Alerted`：已确认敌情，准备进入战斗组织。
6. `Engaged`：明确战斗中，持续追击/攻击玩家。
7. `Searching`：失去目标后在局部搜索。
8. `Returning`：脱战，返回巡逻点或区域锚点。

### 6.2 行为任务层（Task）

描述敌人当前在做什么：

- `Patrol`
- `Wander`
- `GuardPoint`
- `MoveToSound`
- `MoveToLastSeen`
- `ChaseTarget`
- `HoldObjective`
- `Reinforce`
- `ReturnToAnchor`

同一个警戒状态下可以有不同任务，例如：
- `Investigating + MoveToSound`
- `Engaged + Reinforce`
- `Returning + ReturnToAnchor`

### 6.3 异常状态层（Condition）

独立于决策层的异常效果：

- `Burning`
- `Stunned`
- `KnockedBack`
- `Lured`
- `Suppressed`（可选）

这样可以避免旧设计里把“着火”“被引诱”硬塞进主状态机。

## 7. 五类关键触发机制的正式落地方式

以下 5 类机制是新模式遭遇体验的核心，必须成为独立系统，而不是散落在单个怪物类里。

### 7.1 视线触发

用途：
- 作为主要发现方式。
- 支持开阔区暴露、背后潜行、掩体规避。

规则：
- 敌人定期做视锥检测，不是每帧全图扫。
- 检测条件：
  - 距离在视觉范围内
  - 玩家处于视野夹角内
  - 中间无遮挡
  - 玩家姿态与速度可对可见性加权

结果：
- 先进入 `Suspicious` 或 `Alerted`，不是强制瞬间全追击。
- 若同一玩家持续暴露，则进入 `Engaged`。

### 7.2 近距离感应

用途：
- 解决贴脸不发现的蠢行为。
- 保证近身接触有可信度。

规则：
- 玩家进入近距离感应圈，敌人可无视视角直接警觉。
- 玩家奔跑、冲刺、近战、翻越障碍时，感应半径额外增大。
- 蹲伏、低速移动时，感应半径减小。

结果：
- 通常直接从 `Idle` 跳到 `Alerted`。
- 极近距离或受击后可直接进入 `Engaged`。

### 7.3 声音触发

用途：
- 惩罚高调玩法。
- 为武器、爆炸、任务装置提供统一的引怪系统。
- 让“开枪越久，地图越热”具备系统解释。

实现：
- 所有重要事件不直接广播给怪，而是发出 `NoiseEvent`。
- `NoiseSystem` 统一管理：
  - 声源位置
  - 声强
  - 类型
  - 基础半径
  - 墙体衰减
  - 对不同敌种的吸引系数

典型声源：
- 消音枪声：低强度
- 普通枪声：中强度
- 爆炸：高强度
- 任务装置：持续脉冲噪声
- 载具/机械活动：长持续噪声

结果：
- 附近敌人进入 `Investigating`。
- 某些遭遇模板会把噪声视为增援触发条件。
- 区域噪声值上升，推动区域升级。

### 7.4 伤害连锁

用途：
- 单体受击能引发周边局部反应。
- 让玩家偷袭一只怪后，附近集群有合理反馈。

规则：
- 本体受击后立即进入高警戒。
- 同组怪：高概率立即同步进入 `Alerted`。
- 同 Encounter 其他怪：中概率响应。
- 同 Zone 内其他游荡怪：低到中概率响应。
- 相邻 Zone：只在高强度伤害或爆炸时响应。

结果：
- 不再是一个固定半径固定概率广播。
- 传播范围和概率由“关系层级 + 事件强度”共同决定。

### 7.5 同伴提醒

用途：
- 防止敌人彼此完全孤岛。
- 形成局部集群感，但不导致全图瞬时激活。

规则：
- 传播按关系层级进行：
  1. 同 PatrolGroup：高权重
  2. 同 Encounter：中高权重
  3. 同 Zone：中低权重
  4. 跨 Zone：极低权重，仅在高压或特殊事件下生效

传播可以带有延迟：
- 近距离近乎立即生效
- 远距离通过 0.5~2 秒延迟形成“波纹式升温”

## 8. 区域系统设计

### 8.1 Zone 的作用

Zone 不是简单的圆形触发器，而是导演理解地图的中层容器。一个 Zone 至少应包含：

- 空间边界
- 风险等级
- 刷怪源列表
- 任务点列表
- 常驻敌人预算
- 当前警戒值
- 当前遭遇列表
- 冷却状态

### 8.2 Zone 的状态

建议状态：

1. `Dormant`：低活跃或未启用。
2. `Ambient`：维持游荡和巡逻。
3. `Warming`：局部升温，开始向可战斗状态过渡。
4. `Combat`：正在交战，可持续生成遭遇和增援。
5. `Exhausted`：近期打过一场，暂时不再大量刷新。
6. `Cooldown`：冷却期，逐步恢复到 Ambient。

### 8.3 Zone 中的常驻内容

每个 Zone 默认维护：
- 游荡预算
- 巡逻预算
- 特殊怪上限
- 增援入口权重

这样地图上就能始终存在“可探索、可接触、可升级”的局部威胁，而不是完全依赖任务开启。

## 9. 遭遇（Encounter）系统设计

### 9.1 Encounter 的意义

Encounter 是新模式的核心单位。区域只是容器，真正的战斗节奏由 Encounter 决定。

### 9.2 Encounter 类型

建议首批支持以下类型：

1. `AmbientPatrol`：平时巡逻组。
2. `AmbientWanderPack`：区域游荡组。
3. `TriggeredAmbush`：玩家进入触发后从固定点压出。
4. `ObjectiveDefenseWave`：任务防守波次。
5. `DynamicReinforcement`：导演派来的增援。
6. `PursuitPack`：玩家高压状态下的追击组。

### 9.3 Encounter 生命周期

- `Pending`：已排队，尚未刷出。
- `Spawning`：正在从刷怪源分批生成。
- `Active`：战斗进行中。
- `Searching`：失去目标后搜索残余阶段。
- `Resolved`：已清除或超时结束。
- `Cooldown`：记录用于区域冷却与预算回收。

### 9.4 Encounter 和 Zone 的关系

- Zone 决定一个区域允许出现哪些 Encounter 类型。
- Director 决定当前是否要在某个 Zone 创建 Encounter。
- Encounter 决定自己如何刷怪、如何升级、何时结束。

## 10. 刷怪系统设计

### 10.1 刷怪不再直接调用单体生成

新模式禁止在各个逻辑角落直接 `new CommonZombie()` 来实现玩法。

正确流程为：
1. Director 决定要创建 Encounter。
2. Encounter 请求 SpawnGroup。
3. SpawnGroup 选择可用 SpawnSource。
4. SpawnRule 验证当前是否满足刷怪条件。
5. 生成敌人并绑定所属 Encounter / Zone / PatrolGroup。

### 10.2 SpawnSource 分类

建议首批支持：

- `HiddenInterior`：玩家视野外的房屋/巷道/死角
- `EdgeEntry`：地图边缘入口
- `TunnelExit`：洞口/地下设施
- `ObjectiveSource`：任务点专用
- `ReinforcementLane`：专用增援线

### 10.3 SpawnRule 规则建议

每个刷怪源都可配置多个规则：

- 最小玩家距离
- 最大玩家距离
- 必须在视线外
- 必须在指定 Zone 内
- 只能在指定 Encounter 类型使用
- 只能在警戒等级高于某阈值时启用
- 是否允许刷特殊敌人

### 10.4 SpawnGroup 组成

建议用数据定义，而不是硬编码：

- 组名
- 总预算成本
- 敌人列表
- 生成间隔
- 组内角色分配
- 生成后默认任务

例如：

- `ScoutPatrol_A`
  - 3 Common
  - 默认 `Patrol`

- `AlarmResponse_B`
  - 4 Common + 1 Uncommon
  - 默认 `MoveToSound`

- `ObjectiveWave_C`
  - 6 Common + 1 Charger + 2 Armor
  - 默认 `HoldObjective`

## 11. 地图压力与导演预算

### 11.1 两个核心概念

新模式必须区分：

1. **Threat Budget（威胁预算）**：当前允许活跃多少敌人、多少遭遇、多少增援。
2. **Pressure（地图压力）**：玩家当前把地图吵成什么样、热到什么程度。

### 11.2 压力来源

地图压力不应简单等于“时间越久越高”，而是由多因素构成：

- 玩家总停留时间
- 当前任务阶段
- 枪声频率
- 爆炸频率
- 区域内击杀速度
- 同时激活的 Zone 数量
- 是否连续在高危区域推进
- 是否触发报警型装置/任务节点

### 11.3 压力的作用

压力影响：
- 增援出现频率
- 增援组规模
- 特殊敌人权重
- 跨区追击概率
- 巡逻刷新速度
- 任务波次附加强度

### 11.4 预算的作用

预算用于防止失控：
- 全图最大敌人预算
- 单 Zone 最大敌人预算
- 单 Encounter 最大预算
- 同时活跃 Encounter 数量上限

这样可以同时保证体验和性能。

## 12. 任务系统与区域联动

### 12.1 任务不是导演替代品，而是导演放大器

任务本身不应直接承担所有刷怪逻辑。正确方式是：
- 任务只负责提出“我要制造什么压力与目标”。
- Director 和 Encounter 层决定如何编排敌人。

### 12.2 首批任务类型建议

1. 占点防守
2. 装置激活/上传
3. 护送移动目标
4. 清理巢穴/破坏点
5. 搜索与提取

### 12.3 任务与遭遇联动方式

以“装置激活”举例：

1. 玩家到达 Objective Zone。
2. Zone 由 `Ambient` 进入 `Warming`。
3. 玩家启动装置，任务进入运行阶段。
4. Director 在本 Zone 与相邻 Zone 中创建 `ObjectiveDefenseWave` Encounter。
5. 噪声系统持续制造脉冲声源。
6. 压力升高，增援源开放。
7. 任务结束后，Zone 进入 `Exhausted/Cooldown`。

---

## 13. 敌人类型与角色定位

### 13.1 统一框架，参数区分

所有敌人共享同一套：
- 感知输入
- 警戒状态
- 遭遇归属
- 导演调度

不同敌人通过配置区分：
- 生命
- 速度
- 视觉/听觉权重
- 近距感应半径
- 警戒传播强度
- 攻击距离
- 任务倾向（追击/压制/守点/前压）

### 13.2 初始三类角色建议

#### 普通怪（Swarm / Common）
- 主力填充单位
- 负责近身压迫
- 视线和噪声反应中等

#### 重甲怪（Armor / Uncommon）
- 响应速度偏慢
- 更适合出现在守点和任务波次中
- 承担压力峰值时的肉盾职责

#### 冲锋怪（Rush / Charger）
- 少量出现
- 作为波次尖刺或增援突脸单位
- 不适合作为常驻游荡主力

## 14. 数据驱动与可配置性要求

### 14.1 必须数据化的部分

以下内容不允许长期硬编码在逻辑里：

1. 敌人感知参数
2. 区域预算与危险等级
3. 遭遇模板
4. 刷怪组组成
5. 压力曲线
6. 增援冷却与规模
7. 各类噪声事件半径与权重

### 14.2 建议配置对象

#### `EnemyArchetypeConfig`
包含：
- 基础生命/速度/伤害
- 视觉半径
- 听觉半径
- 近距感应半径
- 警戒传播系数
- 脱战时间
- 默认任务倾向

#### `EncounterZoneConfig`
包含：
- Zone 类型
- 常驻预算
- 巡逻模板池
- 增援入口列表
- 冷却时间
- 特殊敌人权重

#### `EncounterTemplateConfig`
包含：
- Encounter 类型
- 触发条件
- SpawnGroup 池
- 最大持续时间
- 升级规则
- 结束条件

#### `DirectorConfig`
包含：
- 全图基础预算
- 人数倍率
- 压力上限
- 增援曲线
- 同时活跃 Zone 上限
- 高压阈值

## 15. 运行时流程

### 15.1 地图启动

1. `EncounterGamemode` 创建。
2. `EncounterDirector` 创建并读取全局配置。
3. 地图中的 Zone、SpawnSource、Objective 节点注册到 Director。
4. Director 根据 Zone 配置，先为部分区域建立 Ambient 巡逻和游荡 Encounter。

### 15.2 玩家进入区域

1. Director 识别玩家进入某个 Zone。
2. Zone 从 `Dormant/Ambient` 提升到 `Warming`。
3. 区域开始增加巡逻更新频率、激活更多刷怪源、提升感知权重。
4. 若玩家继续推进，遭遇被正式创建。

### 15.3 玩家开火或爆炸

1. 产生 `NoiseEvent`。
2. 周边敌人进入 `Suspicious/Investigating`。
3. Zone 的局部警戒值上升。
4. 若超过阈值，Zone 进入 `Combat`。
5. Director 判断是否额外派增援 Encounter。

### 15.4 玩家启动任务

1. 任务改变自身阶段。
2. 向 Director 提交强制升温请求。
3. Director 提高该区域及相邻区域权重。
4. 创建防守或追击 Encounter。
5. 在任务进行中周期性制造噪声和波次调度。

### 15.5 玩家脱战

1. 单体怪物从 `Engaged` 转入 `Searching`。
2. 搜索失败后转入 `Returning`。
3. Encounter 若长时间无有效接触，则标记 `Resolved`。
4. Zone 警戒值下降，转入 `Cooldown`。
5. Director 回收预算。

## 16. 性能与规模控制

### 16.1 预算优先于数量

新模式不能只看“僵尸数量”，必须看预算。原因：
- 特殊怪开销更高
- 不同 Encounter 的场面成本不同
- 区域内是否交战比单纯活着几只怪更重要

### 16.2 Tick 分级

建议：
- 近战中的怪：高频更新
- 可见区域但未交战：中频更新
- 远区 Ambient 怪：低频更新
- `Dormant` 区域：极低频或事件驱动更新

### 16.3 导演更新节奏

Director 不应每帧做全图复杂扫描，应分层：
- 每帧处理轻量状态
- 每 0.2~0.5 秒刷新区域状态
- 每 1~2 秒刷新预算与候选 Encounter
- 大范围重算尽量低频

## 17. 具体实现阶段建议

### Phase 1：模式骨架与导演骨架

目标：跑通新模式入口，不考虑完整内容。

实现：
1. 新建 `EncounterGamemode`。
2. 新建 `EncounterDirector`。
3. 修改 `Game.cs` 的模式装配点，支持新模式启动。
4. 建立 Zone 注册与基础运行框架。

验收：
- 地图可启动到新模式。
- Director 可读取 Zone。
- 不再依赖旧 `SurvivalGamemode + GameDirector`。

### Phase 2：区域与刷怪源

目标：地图能维持基础游荡与巡逻。

实现：
1. 实现 `EncounterZone`。
2. 实现 `SpawnSource / SpawnRule / SpawnGroup`。
3. 让 Director 能在指定 Zone 内创建 Ambient Encounter。

验收：
- 区域内能维持游荡敌人。
- 不同 SpawnSource 会按规则选点。

### Phase 3：单体 Brain 与感知系统

目标：敌人不再无脑全图追。

实现：
1. 实现 `ZombieBrain`。
2. 实现 `PerceptionSystem` 与 `NoiseSystem`。
3. 引入警戒层、任务层、异常层。
4. 将旧 `CommonZombie` 的上层 AI 逐步迁移给 Brain。

验收：
- 视线、近距、声音、受击、同伴提醒都能独立触发。
- 怪可以调查、追击、失目标、回归。

### Phase 4：遭遇模板与任务联动

目标：形成可玩的局部战斗。

实现：
1. 实现 `EncounterInstance`。
2. 实现巡逻、增援、任务波次 3 类模板。
3. 任务可提升区域状态并触发波次。

验收：
- 玩家做事时可触发多波遭遇。
- 常态下地图仍有 Ambient 威胁。

### Phase 5：全图压力与高级导演

目标：形成“越打越热”的地图节奏。

实现：
1. 加入全图压力曲线。
2. 实现高压追击组、跨区增援、特殊怪加权。
3. 完善区域冷却与预算回收。

验收：
- 玩家高调推进时，地图会持续追逼。
- 低调推进时，不会全图同时沸腾。

## 18. 地图制作支持方案

### 18.1 Hammer / 地图标注需求

新模式建议至少支持以下地图实体或组件：

1. `EncounterZoneVolume`：定义区域边界。
2. `EncounterSpawnSource`：定义刷怪源。
3. `EncounterObjectivePoint`：任务点。
4. `EncounterRoute`：巡逻路径或增援路线。
5. `EncounterNoiseEmitter`：持续性噪声点。

### 18.2 地图作者的工作方式

地图作者主要做：
- 划区域
- 标刷怪源
- 设巡逻路线
- 放任务节点
- 标可增援入口

程序负责：
- 何时用
- 用哪个组
- 用多少预算
- 何时升级为战斗

## 19. 对现有代码的迁移策略

### 19.1 迁移原则

- 不在旧模式代码上硬改出新模式。
- 新模式独立成一条运行路径。
- 先接管模式入口，再逐步接管 AI 与刷怪。

### 19.2 推荐迁移顺序

1. 在 `Game.cs` 中把模式装配抽出来。
2. 新模式先能启动空导演。
3. 再接入 Zone 与 SpawnSource。
4. 再把单体 AI 从 `CommonZombie` 中拆出去。
5. 最后再接任务和高压系统。

### 19.3 旧代码处理原则

- 保留旧模式文件，不删除。
- 新模式不直接引用旧 `GameDirector`。
- 旧 `CommonZombie` 的简单追击逻辑只作为迁移过渡参考。
- 之前临时写的原型文件不纳入正式设计，应清理。

## 20. 风险与控制点

### 20.1 最大风险

1. **AI 状态耦合**：如果继续把感知、任务、异常写在一个 enum 里，后期会失控。
2. **刷怪入口分散**：如果很多地方仍然直接实例化怪物，会让导演系统失去控制。
3. **区域职责过重**：Zone 负责太多逻辑会变成新的大杂烩。
4. **压力无预算约束**：只做压力不做预算，很容易敌人数失控。
5. **过早追求高级行为**：包抄、协同、复杂阵型不应放在第一阶段。

### 20.2 控制策略

- 所有敌人出生必须带所属 Encounter。
- 所有 Encounter 都必须消耗预算。
- 所有警戒传播必须经过统一 Alert 层。
- 所有噪声必须经过统一 Noise 层。
- 单体 AI 状态拆层，避免大 enum 污染。

## 21. 最终结论

针对本项目要实现的类潜兵开放地图遭遇战，新模式的正确做法不是继续扩展旧的 `SurvivalGamemode + GameDirector`，也不是把已有底层全部推倒，而是：

**复用底层执行能力，重建上层遭遇战框架。**

具体来说：

1. 复用现有怪物实体、移动、导航、攻击、受击与动画系统。
2. 新建 `EncounterGamemode` 取代旧生存模式作为新玩法入口。
3. 新建 `EncounterDirector` 管理地图压力、预算、区域升温、增援与遭遇调度。
4. 用 `EncounterZone + EncounterInstance + SpawnSource/Rule/Group` 取代旧的单一刷怪导演。
5. 用 `ZombieBrain + Perception/Noise/Alert` 取代旧的简单追击和固定概率连锁。
6. 所有关键参数都走配置，不再长期写死在代码逻辑里。

这套方案既能满足：
- 区域游荡
- 任务多波次
- 全图压力持续逼近
- 非全图即时锁仇恨

也能保留现有工程里最值钱的底层移动与敌人实体资产。

## 22. 建议的下一步

下一步建议不是直接编码全部系统，而是先把实现拆成一个明确的开发计划，至少分为：

1. 模式入口与导演骨架
2. 区域与刷怪源系统
3. AI Brain 与感知系统
4. 遭遇模板与任务联动
5. 压力与高压增援系统

在进入编码前，应先确认：
- Zone 是体积盒、Nav 区块集合，还是 Hammer 自定义实体边界。
- 配置是写 C# 常量、JSON、资源资产，还是 Hammer 属性。
- 首个可玩版本优先做哪 2~3 种 Encounter 模板。

这三点一旦确认，就可以进入正式的实现分解。
